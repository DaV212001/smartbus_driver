# Ticket Validation & Offline Verification — Technical Specification

This document outlines:
1. **Online Ticket Validation API Specification** (NestJS backend API endpoints, payload DTOs, and server response structures).
2. **Offline Validation & Verification Framework in Flutter** (The mathematical and architectural patterns for verifying tickets without cellular/network access on the bus).

---

## 1. Online Ticket Validation API Specification

The driver app calls this endpoint when the bus has an active network connection (online mode). It verifies the cryptographic authenticity of the scanned passenger QR code and updates the ticket status to `USED` in real time.

### 1.1 Validation Request Schema
* **Route:** `POST /tickets/validate`
* **Headers:**
  ```http
  Authorization: Bearer <JWT_ACCESS_TOKEN>
  Content-Type: application/json
  ```
* **Request Body (JSON):**
  ```json
  {
    "qrPayload": "{\"ticketId\":\"c4974f26-0d12-4217-bfbe-d88562d5f04b\",\"passengerId\":\"a38fa721-0072-4d22-b5e1-512c0199e52e\",\"routeId\":\"7df41da8-d421-4b10-85f2-cb8a99db324b\",\"boardingStopId\":\"44ea3d02-cf32-4d55-89f4-b2586b51829e\",\"dropoffStopId\":\"a10d9382-7721-4a11-8fe6-b3334ca6a84f\",\"fareAmount\":12.5,\"expiresAt\":\"2026-05-18T15:10:00.000Z\",\"issuedAt\":\"2026-05-18T14:10:00.000Z\"}",
    "qrSignature": "82a8b9d90ec82ef0e3c15560b371ee0c76db31ab902e88c03fe7e6c43422947a",
    "inspectionMode": false
  }
  ```
  * `qrPayload`: The stringified passenger QR JSON block containing the full ticket details (to prevent tampering).
  * `qrSignature`: A hex-encoded **HMAC-SHA256** signature generated by the backend upon purchase using `QR_SIGNING_SECRET`.
  * `inspectionMode` (optional, default `false`): If `true`, validates signature, expiry, and state but does **not** consume the ticket (i.e. status remains `ACTIVE`).

## 2. Offline Validation Framework in Flutter

When a bus traverses areas with poor cell service (e.g. tunnels or remote areas), the driver's phone will experience network drops. The driver app must continue validating boarding passenger tickets.

```
+------------------+     Decodes Payload &     +--------------------+
|  Passenger QR    | ------------------------> | Driver Flutter App |
| (Data + HMAC)    |                           |                    |
+------------------+                           +--------------------+
                                                         |
                                           +-------------+-------------+
                                           |                           |
                                           v                           v
                                  [ ONLINE MODE ]            [ OFFLINE MODE ]
                                           |                           |
                                           v                           v
                                  Send HTTP POST               Verify Signature
                                 /tickets/validate             Locally via HMAC
                                           |                           |
                                           v                           v
                                     Save Result               Check Local Cache
                                     to DB logs                for Double Boarding
                                                                       |
                                                                       v
                                                               Queue in Database
                                                                 for Batch Sync
```

### 2.1 Cryptographic Verification Principle
Because the backend signs the `qrPayload` string using **HMAC-SHA256** and a shared secret (`QR_SIGNING_SECRET`), the driver application can perform the identical hashing formula locally. 
* The driver app stores the `QR_SIGNING_SECRET` securely (in encrypted storage like `flutter_secure_storage`).
* The app performs `Hmac(sha256, secret).convert(qrPayload)` locally.
* If the resulting hash exactly matches `qrSignature`, the app mathematically proves that the ticket is authentic and was issued by the backend, requiring **zero** internet access.

### 2.2 Flutter Core Step-by-Step Implementation

To build this in Flutter, use the standard `crypto` library (for HMAC-SHA256), a local key-value store like `Hive` or an SQL engine like `sqflite` (for tracking local usage and queueing uploads), and `connectivity_plus` to monitor connection state.

Add these libraries to the `pubspec.yaml`:

#### Step 3: Implement Offline Cryptographic Verifier
Create a service that replicates the backend's signature verification and evaluates local trip-matching rules.
```dart
import 'dart:convert';
import 'package:crypto/crypto.dart';

class LocalTicketValidator {
  final String _sharedSecret;

  LocalTicketValidator(this._sharedSecret);

  /// 1. Mathematically checks if the QR payload hasn't been altered
  bool verifySignature(String payload, String signature) {
    final key = utf8.encode(_sharedSecret);
    final bytes = utf8.encode(payload);
    
    final hmac = Hmac(sha256, key);
    final digest = hmac.convert(bytes);
    
    return digest.toString().toLowerCase() == signature.toLowerCase();
  }

  /// 2. Performs local business logic validation
  OfflineValidationResult validateOffline({
    required String payload,
    required String signature,
    required String activeRouteId,
    required List<String> previouslyScannedTicketIds,
    bool inspectionMode = false,
  }) {
    // A. Verify Cryptographic Integrity
    if (!verifySignature(payload, signature)) {
      return OfflineValidationResult(
        result: 'INVALID_SIGNATURE',
        message: 'Cryptographic signature is invalid.',
      );
    }

    // Parse JSON
    Map<String, dynamic> jsonMap;
    try {
      jsonMap = jsonDecode(payload) as Map<String, dynamic>;
    } catch (_) {
      return OfflineValidationResult(
        result: 'INVALID_SIGNATURE',
        message: 'Malformed QR Payload format.',
      );
    }

    final data = QrPayloadData.fromJson(jsonMap);
    final now = DateTime.now();

    // B. Check Expiry
    if (now.isAfter(data.expiresAt)) {
      return OfflineValidationResult(
        result: 'EXPIRED',
        message: 'This ticket has expired.',
        ticketId: data.ticketId,
      );
    }

    // C. Check Route Matching (Ensures passenger is on the right bus/route)
    if (data.routeId != activeRouteId) {
      return OfflineValidationResult(
        result: 'INVALID_ROUTE',
        message: 'Ticket is issued for a different bus route.',
        ticketId: data.ticketId,
      );
    }

    // D. Local Double Boarding Prevention (Check cache)
    if (!inspectionMode && previouslyScannedTicketIds.contains(data.ticketId)) {
      return OfflineValidationResult(
        result: 'ALREADY_USED',
        message: 'This ticket has already been boarded on this bus.',
        ticketId: data.ticketId,
      );
    }

    // E. Validation Successful
    return OfflineValidationResult(
      result: inspectionMode ? 'INSPECTION_ONLY' : 'VALID',
      message: inspectionMode ? 'Inspection valid.' : 'Ticket validation successful.',
      ticketId: data.ticketId,
      payload: data,
    );
  }
}

class OfflineValidationResult {
  final String result; // 'VALID', 'EXPIRED', 'ALREADY_USED', 'INVALID_SIGNATURE', 'INVALID_ROUTE', 'INSPECTION_ONLY'
  final String message;
  final String? ticketId;
  final QrPayloadData? payload;

  OfflineValidationResult({
    required this.result,
    required this.message,
    this.ticketId,
    this.payload,
  });

  bool get isValid => result == 'VALID' || result == 'INSPECTION_ONLY';
}
```

#### Step 4: Manage Local Usage & Offline Queues
Use **Hive** to hold two essential collections:
1. `scanned_tickets_cache`: Stores a list of all `ticketId` values processed during the active trip (cleared when the trip is ended). Prevents offline double-boarding.
2. `offline_sync_queue`: A reliable FIFO queue of scanned items to upload to the server once internet becomes active.

```dart
import 'package:hive_flutter/hive_flutter.dart';

class LocalQueueService {
  static const String _cacheBoxName = 'scanned_tickets_cache';
  static const String _queueBoxName = 'offline_sync_queue';

  /// Call once inside main.dart initialization
  static Future<void> init() async {
    await Hive.initFlutter();
    await Hive.openBox<String>(_cacheBoxName);
    await Hive.openBox<Map>(_queueBoxName);
  }

  final Box<String> _cacheBox = Hive.box<String>(_cacheBoxName);
  final Box<Map> _queueBox = Hive.box<Map>(_queueBoxName);

  List<String> getScannedTicketIds() {
    return _cacheBox.values.toList();
  }

  Future<void> cacheTicketId(String ticketId) async {
    await _cacheBox.add(ticketId);
  }

  Future<void> clearTripCache() async {
    await _cacheBox.clear();
  }

  /// Queues the scan details to be synced batch-wise to /sync/validations
  Future<void> queueScan({
    required String qrPayload,
    required String qrSignature,
    required String scannedAt,
    required bool inspectionMode,
    required String localResult,
  }) async {
    final Map<String, dynamic> scanRecord = {
      'qrPayload': qrPayload,
      'qrSignature': qrSignature,
      'scannedAt': scannedAt,
      'inspectionMode': inspectionMode,
      'localResult': localResult,
    };
    await _queueBox.add(scanRecord);
  }

  List<Map<String, dynamic>> getQueue() {
    return _queueBox.values.map((item) => Map<String, dynamic>.from(item)).toList();
  }

  Future<void> clearQueueItems(int count) async {
    for (int i = 0; i < count; i++) {
      if (_queueBox.isNotEmpty) {
        await _queueBox.deleteAt(0); // Pop from front
      }
    }
  }
}
```

#### Step 5: Connectivity Listener and Auto-Sync Controller
Uses `connectivity_plus` to detect network status shifts. When network is restored, it pulls all records in `offline_sync_queue` and POSTs them to `/sync/validations` in a batch (up to 100 scans per batch).

```dart
import 'package:connectivity_plus/connectivity_plus.dart';
import 'package:get/get.dart';
import 'package:dio/dio.dart';

class SyncController extends GetxController {
  final LocalQueueService _queueService = LocalQueueService();
  final Connectivity _connectivity = Connectivity();
  final Dio _dio = Dio(); // Configure with base url & authorization interceptors
  
  var isSyncing = false.obs;

  @override
  void onInit() {
    super.onInit();
    // Start listening for connection changes
    _connectivity.onConnectivityChanged.listen((ConnectivityResult result) {
      if (result != ConnectivityResult.none) {
        triggerBatchSync();
      }
    });
  }

  Future<void> triggerBatchSync() async {
    if (isSyncing.value) return;
    
    final queue = _queueService.getQueue();
    if (queue.isEmpty) return;

    isSyncing.value = true;
    
    try {
      // Chunk queue items to maximum of 100 per API request constraint
      final int batchSize = queue.length > 100 ? 100 : queue.length;
      final List<Map<String, dynamic>> batchScans = queue.sublist(0, batchSize);

      final response = await _dio.post(
        '/sync/validations',
        data: {
          'scans': batchScans,
        },
      );

      if (response.statusCode == 200) {
        // Remove uploaded items from local store
        await _queueService.clearQueueItems(batchSize);
        
        // If there are still items left in the queue, recurse sync
        if (_queueService.getQueue().isNotEmpty) {
          await triggerBatchSync();
        }
      }
    } catch (e) {
      print('Offline validation sync failed: $e');
    } finally {
      isSyncing.value = false;
    }
  }
}
```

## 3. Key Architectural Advantages
1. **Zero Latency Boarding:** Even when online, checking cryptographic signatures mathematically *first* on the device guarantees that the app can display an "Invalid Ticket" message instantly, rather than making a network request for a tampered QR code.
2. **Deterministic Security:** HMAC-SHA256 verification relies entirely on solid mathematics. A passenger cannot forge, edit, or copy a ticket payload because altering a single byte makes the local HMAC signature check fail.
3. **Robust Reconciliation:** By saving the precise timestamp when the validation physically occurred on the bus (`scannedAt`), the backend can cleanly resolve transaction timing conflicts (e.g. if the passenger scans online at a station *after* scanning offline in the bus, the backend detects the bus scan happened first and updates the status to match the actual sequence of events).
